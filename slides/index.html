The Problem

Agentic Coding Has an Audience Problem

The Agentic Coding Landscape

General Audiences

  • Vibe-coding culture
  • Prompt-and-pray workflow
  • Zero engineering discipline

Black-Box SDLC

  • Agent does everything
  • Requirements to Code in one shot
  • Human = spectator

Opinionated & Opaque

  • Cannot inspect the plan
  • Cannot steer mid-execution
  • Cannot say “no, not like that”
  • Every week a new tool claims to build your app from a single prompt.
  • But who are these tools actually built for?

The Irony

LLMs Are Great At

  • Syntactically correct code generation
  • API reasoning & pattern matching
  • Working across massive training data

LLMs Are Terrible At

  • Requirements engineering
  • Architectural judgment
  • Knowing when to stop and ask
“They’re brilliant at generating code, but they can’t tell you what to build or why.”

The Insight

“The bottleneck is not code generation.
It’s specification and verification.”

Which led us to ask:

What if we stopped treating the LLM as an autonomous engineer
and started treating it as a disciplined tool in a human-led engineering workflow?
02

The Answer: Corge

Specification-Driven, Engineer-First

What Is Corge?

A specification-driven, software-engineer-first agentic development system. It enforces a controlled engineering workflow where ambiguity decreases at every stage.

The Three Agents

Agent Phase Role
Specification Agent SPEC Elicits requirements, runs Socratic Q&A, produces structured Specification with acceptance criteria
Planning Agent PLAN Generates TechnicalPlan (architecture) and ProceduralSteps (executable checklist)
Coding Agent CODE Executes the plan step-by-step — reads context, calls the LLM, requests human approval for every destructive action

Each agent has a strict responsibility. No overlap. No mission creep.

User Journey

1 Launch & Repo

  • Point Corge at any repository or empty directory

2 Freestyle Canvas

  • Rough ideas with @node: references validated in real time

3 Socratic Wizard

  • Opt-in clarifying questions (capped at 3)

4 Split-Pane Editor

  • Fill [GAP: Topic] templates, then approve

5 Technical Plan

  • Architecture review — edit, approve, or reject (backward nav)

6 Procedural Steps

  • Granular checklist with bracketed IDs

7 Step-by-Step Exec

  • Context hydration → LLM → Approval with live Ctrl+D diffs

8 Verification & Done

  • All criteria checked, tests pass, audit log delivered

Business Value

Traceability

  • Every line of code traced to approved spec and plan step

Safety

  • No write/edit/bash executes without human consent

Privacy

  • No source code or context leaves the machine without user consent

Quality

  • Spec-driven development prevents scope creep and ambiguity

Audit

  • Full argumentation log + audit trail of every tool call and approval

Accountability

  • The human is the engineer; the LLM is the tool

Key Differentiator

  • Not autonomous — collaborative. Human-led, machine-assisted.

Modularity Serves the Business

Swappable UI

  • Replace Textual TUI with Web UI or IDE plugin — zero business logic changes
  • UI module only depends on UiPort protocol; swap it out freely

Swappable Providers

  • Swap DeepSeek for Ollama or OpenAI with zero code changes
  • Single Provider protocol adapter handles all model backends

Contracts Layer

  • typing.Protocol defines every module boundary via port interfaces
  • Enables the numbered execution flow from the module map

Testable in Isolation

  • Every module exposes a Protocol; mock the port, test the logic
  • Integration tests verify contract compliance across boundaries

No Tight Coupling

  • UI has zero import from agent module
  • Agents have zero import from tools or providers
  • Compile-time safety via Protocol structural subtyping
  • No dependencies on other modules, only on Protocols

Parallel Development

  • UI team, agent team, and tooling team work independently behind protocol interfaces
  • Contracts serve as the shared agreement — no coordination overhead per change

  • Swappable: UI is a view, not a controller

CanvasScreen & Sticky Note Validation

Layer 2: Middleware — SessionController

What: The master state machine driving all 3 phases across 12 lifecycle states.

Transition Table (simplified)

  • START → REPOSITORY_SELECTION → REPOSITORY_ANALYSIS
  • SPEC_ENTRY → SPEC_VALIDATION → SPEC_APPROVAL
  • PLAN_GENERATION → PLAN_REVIEW → PLAN_APPROVAL
  • EXECUTION → VERIFICATION → COMPLETION_REVIEW → DONE

Backward navigation on rejection — guaranteed by centralized transition table.

ApprovalGateway & Audit Logger

ApprovalGateway

Single chokepoint for all code-modifying operations. 35 lines. 

class ApprovalGateway:
    def approve(self, request) -> ApprovalDecision:
        if request.action == ToolAction.READ:
            return APPROVED
        decision = self._ui.request_approval(request)
        self._audit_logger.record_approval(request, decision)
        return decision

Audit Logger

Dual-stream logging:

  • Local: .agent/audit.jsonl — tool-level detail
  • Global: ~/.config/corge/global_audit.jsonl — cross-project milestones

Safety by Construction

  • Read auto-approves
  • Write/Edit/Bash blocks for human consent
  • Live Ctrl+D code diffs before any action
  • Every decision is audited

Self-Improving Agents

The system learns from every session. After each project, a batch-phase heuristic updater analyzes the argumentation log and adjusts the spec wizard’s behavior for future sessions.

How It Works

  • Records every Socratic Q&A interaction
  • Tracks which questions users override or skip
  • Adjusts question priority for next session
  • No manual tuning — self-calibrating

Abandonment Protection

  • If a user quits mid-spec, the system penalizes its engagement prior
  • Prevents over-asking in future sessions
  • Single bad session can only degrade the model slightly

Conservative Drift Protection

  • A single session can only shift behavior by a limited amount
  • Overrides require multiple consistent sessions to shift behavior
  • After 20 successful sessions with zero overrides, the wizard converges to a stable, minimal-intervention cadence

Key Takeaway

  • The spec wizard gets better over time without human reconfiguration
  • Self-improvement through usage data, not hardcoded rules

Workflow State Chaining & Context Isolation

Step-to-Step State Machine

After each execution step, the system captures what the agent proposed and how the human responded. This state is injected into the next step’s prompt — the agent learns from its corrections mid-session.

3-Layer Context Isolation

Each phase gets only the context it needs. Instruction pollution is prevented by design.

Phase Included Omitted
Spec Canvas, schema, Q&A File lists, AST, code
Plan Spec, repo tree, AST, config Execution logs, tool outputs
Code Spec, plan, graph, memory, artifacts Canvas drafts, Q&A details

Layer 4: Data — Knowledge Graph

SQLite-backed AST parser building a nodes + edges graph of the repository.

Query Grammar

  • files — List all files
  • classes:<path> — Classes in a file
  • imports:<path> — Dependencies
  • imported_by:<path> — Reverse deps
  • node:<node_id> — Specific node

The LLM cannot navigate a filesystem — the knowledge graph gives it a queryable representation.

CREATE TABLE nodes (
    node_id TEXT PRIMARY KEY,
    kind    TEXT NOT NULL,
    path    TEXT NOT NULL,
    name    TEXT NOT NULL
);
CREATE TABLE edges (
    src TEXT NOT NULL,
    rel TEXT NOT NULL,
    dst TEXT NOT NULL,
    PRIMARY KEY (src, rel, dst)
);

Layer 5: Infrastructure

Provider Adapter

Single OpenAI-compatible adapter for supported provider families like:

  • DeepSeek — deepseek-* model prefix
  • OpenAI — Standard endpoint
  • Ollama server — Local API
  • OpenRouter — sk-or-* key prefix

Tool Runtime

Four stateless primitives with safety validation:

  • read — File reading
  • write — File writing
  • edit — Exact occurrence-count matching
  • bash — Sandboxed, no sudo/traversal

Budget Manager

Unconditional transcript compaction:

  • Duplicate removal
  • Large-string clipping
  • Context window ceiling enforcement
  • LLM sees most relevant context

Lessons Learned

From Ideation to Architecture

7 Lessons

1 Specs-first sounds slow; it's faster.

The Socratic wizard catches ambiguities before a line of code is written.

2 Protocol-driven modularity works.

typing.Protocol enabled parallel development across teams.

3 Hardest part is context engineering.

Getting the right info into the prompt, within token limits.

4 Human-in-the-loop is a quality feature.

Every rejection teaches the system. The heuristic updater captures this.

5 Keyboard-first TUI is worth the effort.

Engineers live in the terminal. Corge feels native.

6 YAGNI

Every deferred feature has a named ceiling and upgrade path.

7 Modularity is key

Modular software is not just for POCs.
